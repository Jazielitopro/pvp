# Starter Prompt — Parryhook Build Sessions

Use this prompt at the start of each new build session to give Claude full context.

---

## Prompt Template

```
We are building a competitive PvP arena shooter in Roblox Studio using Luau.

## Game Identity
- First-person, skill-based, high mechanical ceiling
- Inspired by Ultrakill pacing, adapted for multiplayer
- NOT casual — precision and movement mastery are the core skill expression

## Core Design Decisions
- Projectile-based weapons (no hitscan) — bullets have travel time and require prediction
- Revolver: 6 bullets, 3 hits to kill, manual reload, no spread
- Passive health regen after delay — encourages disengage and repositioning
- Dash: no invulnerability frames, cooldown-based, repositioning tool only
- Grapple: attaches to environment only (MVP)
- Parry: deflects/reflects projectiles on precise timing — high risk, high reward
- Perks: all have trade-offs, no dominant strategy allowed
- Progression: cosmetic only, never affects gameplay stats

## Architecture Principles
- Server is authoritative — no client-side damage, ammo, or hit validation
- Client runs ahead visually for responsiveness; server corrects on conflict
- All systems are modular — gameplay logic, visual feedback, and network logic are separated
- Avoid exploit-prone patterns

## Projectile System Modules (designed)
- ProjectileRegistry — server, source of truth for all active projectiles
- ProjectileSpawner — validates fire requests, registers projectiles
- ProjectileSimulator — server (authoritative) + client (visual only)
- CollisionHandler — server, routes hits to damage or parry
- DamageApplicator — server only, applies chunk damage
- ParryHandler — server validation + client input detection
- FeedbackDispatcher — server to client signals for VFX/SFX

## Key Decisions Made

### Lag Compensation — Option C (Rewind with cap)
- Server keeps rolling position history per player (~200ms window)
- On hit validation: rewind target position to client fire timestamp
- Hard cap at 150ms — beyond that, use current server position
- Risk to monitor: artificial ping inflation

### Parry Timing — Two-stage
- MVP (Phase 5 initial): Option B — server checks projectile proximity at input time, generous radius
- Polish (Phase 5 final): Option C — server rewinds projectile to client input timestamp, tight calibrated buffer
- Migration is contained inside ParryHandler, no other module changes needed

## Current Phase
[FILL IN: e.g., "Phase 1 — Movement Prototype, Step 2: Dash implementation"]

## Current Task
[FILL IN: e.g., "Implement the dash mechanic — no iframes, server-side cooldown, cooldown duration TBD"]

## Constraints for This Session
- Build only what is needed for the current step
- Do not add features beyond the current phase
- Flag any edge case or risk you identify before writing code
- Ask before making architectural decisions not covered above
```

---

## How to Use This

1. Copy the prompt above
2. Fill in **Current Phase** and **Current Task** from `build-plan.md.txt`
3. Paste at the start of a new Claude session
4. Claude will have full context and build only what is needed
